# 地区与行业筛选默认规则

当需求出现“地区 / 区域 / 所在地 / 注册地 / 办公地”等企业地域筛选，或出现“行业 / 所属行业 / 国标行业”等行业筛选时，以下规则为默认行为。用户无需额外说明。

## 行政地区

- 默认使用三级级联：省 → 市 → 区（县）。不以三个彼此独立的下拉框替代。
- 选项数据由 `options` 或等价的本地配置注入；使用稳定的行政区划 `code` 作为提交值，`label` 仅用于展示。
- 选择上级后加载或显示下级；更改省必须清空已选市、区，更改市必须清空已选区。
- 筛选摘要、已选标签和请求/本地过滤条件同时保留完整路径与 code，例如“广东省 / 深圳市 / 南山区”。
- 独立输出不得硬编码线上接口、地图 SDK 或宿主项目的地区 store。预览使用最小的三级 mock 选项树即可。

## 国标行业

- 默认使用国民经济行业分类的级联树：门类 → 大类 → 中类 → 小类；不得以扁平行业文本框替代。
- 行业选项同样以 `code` 作为稳定值、以 `label` 作为展示文本；父级变更必须清空所有已选子级。
- 预览和 mock 数据应至少覆盖两条不同门类的完整路径，展示真实的级联、清空与筛选结果变化。
- 不绑定原项目接口、枚举模块或权限逻辑。数据源由组件 prop / 本地常量提供，真实项目接入时可替换。

## 组件接口与覆盖

- 可实现为通用 `RegionCascader`、`IndustryCascader`，或在现有通用级联组件上封装语义化配置；接口至少支持 `value`、`options`、`clearable` 和 change 事件。
- 默认模式不可要求调用方传入复杂配置：地区即三级省市区，行业即国标四级。仅当用户明确要求时，才通过 `level`、`allowIntermediate`、`standard` 等 props 改为省级、城市级、简化行业或其他分类体系。
- 在 Element UI 中优先使用 `el-cascader`；展示值采用完整路径，避免只显示末级名称而失去上下文。

## 验收

- 地区和行业均可清空，且清空后结果恢复为全量 mock 数据。
- 切换任一父级不会残留旧子级值；筛选条件、已选标签、列表/统计结果保持一致。
- 组件在无 options、加载中和无匹配结果时仍保持可用的空态与禁用/提示反馈。
